OpenRAL — Observability & Traceability Report

Branch: claude/observability-traceability-report-d8a5j8  ·  Date: 2026-06-21  ·  Scope: all 8 layers (HAL → Observability) + cross-cutting trace spine  ·  Method: source-grounded audit (every claim carries a file:line reference)
Headline. OpenRAL's observability layer is production-grade and unusually complete — a full OpenTelemetry pipeline (traces + metrics + logs), W3C context propagation across Python ↔ ROS 2 ↔ C++, a typed failure bus, a self-hosted live dashboard, and end-to-end trace ↔ dataset ↔ eval correlation. The principal gap is provenance: rSkill weight signing/verification is specified but not implemented (ADR-0006), so the §8 "replayable from the trace alone" guarantee is forensic, not cryptographically verifiable. Secondary gaps are scoped, not structural: multiprocess log correlation, the Tier-C critic producer, and richer quantization/model-lineage provenance fields.

1Executive summary

OpenRAL treats observability as a first-class layer (Layer 7, python/openral_observability/), not an afterthought. Two CLAUDE.md operating principles set the bar this report measures against: §4 Explicit-beats-implicit ("replanning, dispatcher fallback, quantization, and license posture must show up in logs/traces") and §8 Reproducibility ("a skill execution must be replayable from the trace alone — weights revision pinned, prompts logged, sensor frames captured").

What's strong

  • Full OTel pipeline. Idempotent SDK bring-up wires a TracerProvider + MeterProvider + LoggerProvider, no-op until OTEL_EXPORTER_OTLP_ENDPOINT is set (_sdk.py:107), with head-based sampling and batch tuning via env.
  • Domain span helpers. rskill_span, inference_span, safety_span, reasoner_span (tracing.py:40–187) + a cli.command root span (cli.py:52) seed a single trace per CLI invocation.
  • Cross-process trace context. W3C traceparent inject/extract (propagation.py:37–115) flows the same trace_id through ROS IDL fields into the C++ safety kernel.
  • Typed failure bus + diagnostics. Rate-limited FailureBusPublisher (failure_bus.py:213) on six namespaced topics, plus a 1 Hz DiagnosticsHeartbeat (diagnostics.py:49).
  • Self-hosted dashboard. Embedded OTLP/HTTP receiver, SSE stream, bounded trace index, offline voice prompt (dashboard/app.py, dashboard/store.py:283) — no external Jaeger/Tempo required.
  • End-to-end correlation. trace_id threads execution → dataset rows (frame_trace.py:28) → eval JSON (RSkillEvalResult.trace_id, schemas.py:4672) → replay timeline (replay/correlator.py:95).
  • Real tests, no mocks. ~2,700 LOC of tests over in-memory OTel exporters and real .mcap fixtures, honoring CLAUDE.md §1.11.

What's missing or partial

  • Provenance signing (P0). Sigstore signing/verification from ADR-0006 is unimplemented — no signature manifest field, no verify-on-load; loader only emits an rskill.unverified_provenance warning (loader.py:748).
  • Multiprocess log correlation (P1). The structlog→OTel bridge is single-process (logging.py:47); dispatcher/fleet workers aren't wired.
  • Tier-C critic producer (P1). The /openral/failure/critic contract exists, but no default progress watchdog publishes to it.
  • Provenance depth (P2). Quantization is a boolean flag (no bits/algo); no model-family/lineage field on the execution path; prompt history relies on bag capture rather than a first-class archive.
  • Layer coverage drift (P2). WAM observability is protocol-only (adapters pending v0.3); C++ safety-kernel LTTng + sim/HIL gates pending; spatial-memory persistence Phase 5 pending.

3Per-layer observability & traceability inventory

The eight architectural layers (CLAUDE.md §3). For each: what telemetry/traceability exists today, where it lives, and the layer-local gaps.

Layer 0

HAL — hardware abstraction

shipped
Joint/actuator I/O instrumented at the hot-path boundary; per-call timing and proprio snapshots flow onto spans. E-stop events are counted, not just logged.
Exists
  • Read/send timing histograms: openral.hal.read_state.duration, openral.hal.send_action.duration (semconv.py:218–219); E-stop counter openral.hal.estop.count (semconv.py:220).
  • Hot-path attribute encoders flatten joint state and the next-applied action onto spans with cardinality guards (≤64 joints, ≤8 EEs, 3-dp rounding, ±inf clamp): producer.record_joint_state, record_action, record_ee_poses (producer.py).
  • Multithreaded executor + proprio snapshot timing per ADR-0049.
Gaps
  • No HAL-side latency budget on the read→infer→write round-trip exposed as a metric (tick-level budget exists; per-HAL does not).
Layer 1

Sensors

shipped
Frame freshness is treated as a safety-relevant signal: age and staleness are both metric and span attributes, and stale reads raise ROSPerceptionStale.
Exists
  • Frame age histogram openral.sensors.age_ms + stale-read counter openral.sensors.stale_reads (semconv.py:221–222).
  • Per-frame span attrs (modality/encoding/W×H×C/age + JPEG thumbnail for the dashboard): producer.record_sensor_frame_attrs, encode_frame_thumbnail (producer.py); attribute keys at semconv.py:115–121.
  • SensorFrame captures the raw payload for replay — inline data, a topic reference, or an opaque handle (schemas.py:1716).
Gaps
  • Thumbnail capture is dashboard-oriented; full-resolution frame replay depends on bag/topic capture, not the trace itself.
Layer 2

World State

shipped
Aggregated state carries per-component staleness and a latched-error flag; the scene graph emits its own span for spatial debugging.
Exists
  • Staleness histogram + stale-component up-down counter + latched-error attr: openral.world_state.staleness_ms, .components_stale, .has_latched_error (semconv.py:135–138, 223–224).
  • world.scene_objects span with object list/count/frame/source (semconv.py:144–147), rendered as a dashboard card + SLAM-map markers (ADR-0038/0039).
  • WorldStateStamped IDL carries trace_id, per-component staleness, and (ADR-0035) detected-object parallel arrays; published fast (30 Hz) / slow (5 Hz).
Gaps
  • Spatial-memory persistence is Phase 4 (in-memory + CLIP embed); durable sqlite-vec store + ROS feeder are Phase 5 (planned).
Layer 3

rSkill (S1 fast policy)

shipped
Skill lifecycle and each VLA chunk inference are spanned; tick records are serializable and carry their own trace context for offline join. Weights are revision-pinned.
Exists
  • rskill_span (configure/activate/execute) and inference_span (foreground vs prefetch, chunk index) — tracing.py:40, 70.
  • TickResult with per-stage timings + trace_context (full W3C traceparent persisted for offline replay, schemas.py:5730); RunResult.trace_id links the run to its root span (schemas.py:5761).
  • Revision pinning: InstalledRSkillEntry.revision stores the HF commit SHA (loader.py:128); from_pretrained(..., revision=...) (loader.py:181).
  • License/posture guard at load (non-commercial gate, unsafe-pickle gate, remote-code gate): _check_provenance (loader.py:748).
Gaps
  • Quantization surfaces only as a boolean info.quantized — bits/algorithm/schedule are not standardized on the inference span.
  • No model-family/lineage field on the execution path (present only in RSkillEvalSource, schemas.py:4570).
  • Weight signature verification absent — see P0 below.
Layer 4

Reasoning (S2 LLM supervisor)

shipped
Every reasoner pass is a span with model/tool/tier/error attribution; tool calls are typed (no free-form JSON), and the bounded replanning ladder is observable per failure kind.
Exists
  • reasoner_span per ReasonerCore.tick() with tick_idx, model, force, and (ADR-0018 amendment) reasoner.tier (tracing.py:132).
  • ReasonerTickResult records tool_call, error, elapsed_s, suppressed_reason and the active traceparent (core.py:43).
  • Typed tool palette: ReasonerToolCall discriminated union validated via TypeAdapter before dispatch (tool_use.py:225, 412).
  • Bounded ladder: per-kind retry cap + 100 ms min-interval gate, both surfaced as suppressed_reason values (core.py:43–90).
Gaps
  • Prompts flow over /openral/prompt and onto spans, but there is no first-class prompt archive/playback — auditing prompt history depends on bag capture.
Layer 5

World Action Model (WAM)

partial
The observability contract is defined (failure source + evidence), but concrete adapters that would emit rollout spans are not yet shipped.
Exists
  • Failure source /openral/failure/wam with WamEvidence (horizon, error) in the failure-bus evidence union; WorldModel Protocol + NullWorldModel.
Gaps
  • No wam.rollout span emitter shipped; Cosmos / UnifoLM-WMA-0 / IRASim adapters are v0.3 (planned). Mental-simulation gating is therefore not yet traced.
Layer 6

Safety

shipped kernel gates partial
Safety checks are spanned and severity-counted; the C++ kernel re-parents under the Python tick via W3C context, and violations raise typed exceptions for the supervisor incident log.
Exists
  • safety_span (check_name, severity ∈ info/warn/violation) — tracing.py:98; violation/clamp counters openral.safety.violations, .clamps (semconv.py:216–217).
  • C++ kernel resumes the trace from the traceparent packed in ActionChunk via extract_traceparent (propagation.py:82).
  • Failure source /openral/failure/safety is the Tier-A preempting signal (ADR-0018 amendment); ROSSafetyViolation drives the structured incident log at the supervisor boundary (CLAUDE.md §5).
Gaps
  • Kernel-side LTTng tracepoints + sim/HIL gating are in-flight (ADR-0020); kernel timing breakdown not yet a first-class metric.
Layer 7

Observability (the platform itself)

shipped
python/openral_observability/ + the python/openral_dataset/ bridge. The richest layer in the repo; details in §4 (spine) and §8 (appendix).
Exists
  • SDK lifecycle (_sdk.py:107, 280), span helpers (tracing.py), 28 metric instruments (metrics.py/semconv.py:211), propagation (propagation.py), structlog bridge (logging.py), failure bus (failure_bus.py), diagnostics (diagnostics.py), lifecycle error logging (lifecycle.py:39), system metrics sampler (system_metrics.py:45), LTTng (tracing_lttng.py), CLI root span (cli.py:52).
  • Dashboard (dashboard/app.py, store.py:283) and replay (replay/bag_reader.py:144, correlator.py:95, trace_query.py).
  • Dataset bridge: RolloutRecorder (recorder.py:167), Rosbag2Sink (bag.py:143), LeRobotDatasetSink (sinks.py:93), Rosbag2ToLeRobotConverter (converter.py:96), read_frame_trace (frame_trace.py:28).

  • No export-failure alerting beyond the openral.observability.export_failures counter (semconv.py:225) — the counter exists but nothing acts on it.

4Cross-cutting traceability spine

The mechanisms that make a single trace_id meaningful from CLI invocation all the way to a training-dataset row.

OTel SDK, exporters & sampling

configure_observability() (_sdk.py:107) is idempotent and a no-op until OTEL_EXPORTER_OTLP_ENDPOINT is set (CLAUDE.md §1.12-friendly for CI). It supports OTLP/gRPC (default) and HTTP/protobuf (for the dashboard receiver), head-based sampling via OPENRAL_OTEL_SAMPLE_RATIO (ParentBased + TraceIdRatio, ADR-0010), and batch tuning via OPENRAL_OTEL_SPAN_SCHEDULE_DELAY_MS (~33 Hz default) / OPENRAL_OTEL_METRIC_INTERVAL_MS. shutdown_observability() (_sdk.py:280) drains all three pipelines via an atexit hook.

W3C context propagation (Python ↔ ROS 2 ↔ C++)

The truth is the OTel context; ROS message fields are populated from it (ADR-0018 §6). current_traceparent() (propagation.py:37) reads the active span; inject_traceparent() / extract_traceparent() (propagation.py:52, 82) move it across the wire. The same trace_id appears on ActionChunk, FailureTrigger, PromptStamped, and WorldStateStamped, so a single trace spans the fast loop, the failure bus, and the C++ kernel.

Structured logging

trace_context_processor() stamps trace_id/span_id on every log record (logging.py:31); install_structlog_bridge() routes structlog → stdlib → OTel LoggerProvider → OTLP (logging.py:47). Logs land in the dashboard event log alongside the spans that produced them.

Live dashboard

A self-contained FastAPI app (dashboard/app.py) hosts the OTLP/HTTP receiver (/v1/traces · /v1/metrics · /v1/logs), an SSE live stream (/api/stream), a bounded trace index (/api/traces, /api/spans/{trace_id} — F7), and an offline voice-prompt path (local faster-whisper). The TelemetryStore (store.py:283) is a thread-safe, ring-buffered aggregator (bounded events/metrics/traces). No external collector is required (ADR-0017).

Record / replay

read_bag() (replay/bag_reader.py:144) iterates .mcap without a rosbag2_py dependency, extracting traceparent from both JSON-schema payloads and CDR bytes. build_timeline() (correlator.py:95) query-joins bag messages with dashboard spans by trace_id into a chronological timeline. Replay is forensic — it reconstructs causality, not bit-identical re-execution (see §5).

Dataset ↔ trace ↔ eval correlation

RolloutRecorder (recorder.py:167) stamps each frame with the active tick span's (trace_id, span_id); LeRobotDatasetSink (sinks.py:93) writes them as parquet columns and a meta/openral_traces.json episode→trace map (ISSUE-109). read_frame_trace() (frame_trace.py:28) pivots any (episode, frame) back to its trace. RSkillEvalResult.trace_id (schemas.py:4672) closes the loop from a benchmark number to the trace tree that produced it.

Replay-determinism caveat (document, don't "fix")

Bit-identical re-execution is not a goal: GPU kernels/quantization are non-deterministic and diffusion-based VLAs intentionally sample. The trace/bag are a forensic record (what happened, with full causality), not a deterministic-replay engine. This should be stated explicitly in the glossary and roadmap so the §8 guarantee is not over-read.
